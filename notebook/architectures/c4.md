## Style: C4 Model (Context · Container · Component · Deployment)

Produce all **four C4 levels of the same system** in **one** output. Simon Brown's whole point is that the C4 levels are meant to be read together — Context zooms out to the world around the SUT, Container looks inside, Component zooms further into one Container, Deployment shows where the Containers physically run. Cross-view consistency is the contract: the `payment-service` container shown in the Container view MUST be the same element zoomed into in the Component view and deployed in the Deployment view.

Set `kind: "c4"` and populate **one set** of `elements`, `relations`, `groups`. Tag each item via the `views` array with the level(s) it appears in.

---

## The four views and what goes in each

### View 1 — `c4-context` (Level 1: System Context)

Audience: non-technical executives, sponsors. Answers "who uses this system and what does it depend on?".

Populate:
- `person` — human personas / user types (Retail Customer, Compliance Officer, Front-Office Trader). 2–5 max.
- `system` — exactly **one**: the system under test (the SUT).
- `external-system` — collaborators outside your boundary (KYC vendor, payment network, regulator gateway, market-data feed). 3–8.
- One `group` of `kind: "system-boundary"` surrounding the SUT.

These elements typically also appear in Container view (`views: ["c4-context", "c4-container"]`) — the same persons / external systems are reused.

### View 2 — `c4-container` (Level 2: Container)

Audience: software architects, principal engineers. Answers "what are the deployable units inside the SUT?".

Populate:
- All `person` and `external-system` elements from Context (re-use, don't duplicate — add `c4-container` to their `views` array).
- The `system` SUT element (with `c4-container` added).
- `container` elements — each a separately deployable / runnable unit: web app, API service, worker, database, queue, cache. **Every container has a `technology` tag** ("TypeScript / React SPA", "Java 21 / Spring Boot 3.3", "PostgreSQL 15", "Apache Kafka"). Use `parent` to nest them inside the system-boundary group. 4–10 containers.

### View 3 — `c4-component` (Level 3: Component)

Audience: engineers building inside one Container. Answers "how does one Container decompose into modules?".

Pick **one focal container** from the Container view and zoom in. Populate:
- The focal `container` element (with `c4-component` added to its `views`).
- `component` elements inside it via `parent: <container-id>`. Each is a module / package / class with a single responsibility. Tag with framework-level `technology` ("Spring `@RestController`", "Spring Data JPA repository", "Hexagonal port"). 5–12 components.
- External collaborators the focal container talks to (other containers or external systems) appear here too with `views: [..., "c4-component"]` — engineers need to see what's at the boundary.

If the source covers multiple containers worth zooming into, pick the one with the richest internal structure.

### View 4 — `deployment` (Deployment)

Audience: platform engineering, SRE, security architecture. Answers "where do the Containers physically run?".

Populate:
- `deployment-node` elements forming a nested infrastructure tree: Cloud → Region → VPC / Cluster → Compute unit. Use `parent` to nest them. Tag each with `technology` ("AWS eu-central-1", "Kubernetes 1.30", "EKS", "RDS"). Tag with `views: ["deployment"]`.
- `container` elements from View 2 placed onto leaf deployment-nodes via `parent`. The same container element appears in `c4-container`, `c4-component` (if focal), AND `deployment` — so its `views` array lists all three.
- `external-system` elements for managed-service endpoints the deployment depends on (Cloudflare, managed databases, SaaS, identity providers).

If the source doesn't describe DR / multi-region, do not invent it — that is a hard `inferred: true` flag because DR posture is a business decision.

---

## Cross-view consistency rules (non-negotiable)

1. **One canonical id per logical thing.** If `payment-service` is a container, it has one id (`payment_service`) used by every view that mentions it. Its `views` array lists `["c4-container", "c4-component", "deployment"]` (and possibly `"c4-context"` if the SUT IS just this service).
2. **Persons and external systems re-appear across views.** A Customer in Context view is the same Customer in Container view — list both views on the element, do not create two separate person elements.
3. **`parent` chains must resolve consistently.** A component's `parent` is a container id (which also exists in this output). A container's `parent` is either the SUT id or a deployment-node id, depending on view. Pick one canonical parent — typically the SUT for C4 nesting, OR a deployment-node when modelling deployment. Choose the **most informative** parent and let the renderer hide it when irrelevant for the current view.
4. **Relations carry `views[]` too.** A "writes to" relation between `payment-service` and `core-banking-db` belongs in `c4-container` AND `c4-component` (when the component view is for `payment-service`) AND `deployment`.

---

## Design Discipline (applies to every view)

1. **Pick an architectural pattern that fits the constraints from the sources**: modular monolith, microservices, event-driven / CQRS, hexagonal, layered. The shape of the Container view reflects the choice.
2. **Fill gaps with credible engineering judgment.** A senior architect never produces an "empty waiting to be filled" diagram. Mark every uised element with `inferred: true`.
3. **Be credible about technology.** When sources don't pin a stack, choose conventional current tech (PostgreSQL, Redis, Kafka, S3, Cloudflare, Spring Boot, FastAPI, Node + TypeScript). No "some database", no "messaging system".
4. **Compactness.** Total across all four views: **15–40 elements**. If the system is bigger, drop the Component view scope to a single container.
5. **Stable, descriptive ids.** Short snake_case: `customer`, `payment_service`, `core_banking_db`, `aws_eu_central_1`, `production_cluster`.
6. **Relations carry semantic weight.** Verb-phrase labels ("publishes events to", "writes audit log to", "queries customer master from") and a `technology` tag where the call crosses a transport boundary ("HTTPS/JSON", "AMQP", "JDBC", "gRPC + mTLS").

## Output Format (this pass)

Your final response is the **structured English draft** of the architecture — the next pass converts it to JSON. Include the four views in a single document so the model has all of them in mind at once.

```
# Title
<system name>

## Summary
<2–4 sentences: what the system is, who uses it, the architectural pattern>

## Kind
c4

## Elements
- <element-id> [<element-kind>] (views: <comma-separated views>) — <label>
  technology: <if applicable>
  parent: <if applicable — must reference another element or group>
  description: <one-liner>
  stereotype: <optional — <<datastore>>, <<queue>>>
  inferred: <true if synthesised>
- ...

## Groups
- <group-id> [<system-boundary|deployment-group>] (views: <comma-separated>) — <label>
- ...

## Relations
- <source-id> → <target-id> [<relation-kind>] (views: <comma-separated>) — <label>
  technology: <if applicable>
  inferred: <true if synthesised>
- ...
```

## Quality checklist before returning

- [ ] Every element has a non-empty `views` array.
- [ ] Persons and external systems appear in **multiple** views, not duplicated.
- [ ] The Container view has 4–10 containers; every container has a `technology` tag.
- [ ] The Component view zooms into exactly one focal container.
- [ ] The Deployment view has a nested deployment-node tree AND places containers onto leaves.
- [ ] No `entity` elements (that's the data catalog, not architecture).
- [ ] No `fk-*` relations (that's the data catalog).
- [ ] Every synthesised element is marked `inferred: true`.

Do not output JSON in this pass — output the structured English draft above. **Design freely; mark what you designed.**
