## Required structure

Compose the canvas from four regions. Stack them vertically, split into a 12-column grid, or use a 60/40 column split ‚Äî but **every region must be present**.

### 1 ¬∑ Header strip ‚Äî top, ‚â§ 15% of height (‚â§ 160 px)
- **Title** ‚Äî an *insight headline*, not a topic. ‚â§ 60 chars ideal, 80 hard cap. 56‚Äì80 px, 700‚Äì800 weight. One line, two max.
- **Subtitle** ‚Äî context: scope + date + audience. 28‚Äì36 px, 400‚Äì500 weight, muted color.
- Optional: a small tag/chip (e.g. `CONFIDENTIAL`, `FY24 Q3 BRIEF`, `INTERNAL ‚Äî RISK & COMPLIANCE`) top-right.

### 2 ¬∑ TL;DR ribbon ‚Äî one row directly under the header, ~6‚Äì8% of height
- **One sentence** of "if you read nothing else, read this." 28‚Äì34 px, bold or panel-boxed.
- Lead with the conclusion, not the setup. ("Q3 ACV is 8% behind plan, driven by EMEA enterprise slippage.")
- This is the single most important line on the page.

### 3 ¬∑ Body ‚Äî the dense middle, ~60‚Äì65% of height. Pick **one** body layout:

  **a ¬∑ Findings + visual (default ‚Äî use this unless content demands otherwise)**
  - Left column ~45‚Äì50% width: **3‚Äì5 key findings**, one per card. Each card =
    label (small caps, 20‚Äì22 px) ‚Üí figure or finding (40‚Äì64 px, bold) ‚Üí one-sentence so-what (22‚Äì26 px).
  - Right column ~50‚Äì55% width: **one anchor visual** that dominates the right half:
    a chart (bar / donut / line / waterfall), a KPI grid (3‚Äì4 stat cards), a 2√ó2 matrix, or a process flow.
    Title the visual with a 24‚Äì28 px caption that states the takeaway.

  **b ¬∑ Magazine layout**
  - Hero panel at top of body (~40% body height): one big chart, image, or hero stat.
  - Two- or three-column text block below: 3‚Äì6 short findings, each with a bold lead + 1-line elaboration.

  **c ¬∑ Framework / process one-pager**
  - Centerpiece is a horizontal stepped flow, a stacked maturity model, a 2√ó2 matrix, or a chevron process ‚Äî taking ~70% of body width.
  - Sidebar (~30%) holds 2‚Äì4 marginal callouts: risks, controls, dependencies, or owners.

Regardless of which body layout you pick: **one focal point** in the body. The hero visual or the largest finding card is what the eye lands on first.

### 4 ¬∑ Footer strip ‚Äî bottom, ‚â§ 12% of height
- **Recommendation / "So what"** ‚Äî 1‚Äì3 imperative bullets prefixed with `‚Üí`. 22‚Äì28 px. ("‚Üí Reforecast EMEA pipeline at $11M for Q4." / "‚Üí Escalate two amber controls to Risk Committee Nov 15.")
- **Source citations** ‚Äî small caption, 18‚Äì22 px, muted, right-aligned. Inline cite (e.g. `Sources: 10-K ¬ß2.3 ¬∑ Internal BI ¬∑ Risk Register #4421`).

## Rules specific to one-pagers

- **Density is the goal, not a side effect.** Pack the canvas evenly; aim for ~70‚Äì80% region fill. Generous margins (80‚Äì96 px outer) but no large empty zones.
- **No filler.** Every region carries real, sourced content. No "lorem", no "TBD", no decorative third bullet you invented to balance the grid.
- **No multi-slide rules apply.** Disregard archetype rotation, the 8‚Äì12 slide arc, title-slide patterns, section dividers ‚Äî none of that is relevant here.
- **Numbers ground the brief.** Cite real figures with units; round for legibility (e.g. `$128M`, `+38%`, `7.4√ó`). Use `font-variant-numeric: tabular-nums` everywhere digits align.
- **Visual hierarchy is brutal.** Title 3‚Äì5√ó body. Hero figure 6‚Äì10√ó body. Subtitle and footer caption sit in a muted color so they don't compete with the body.
- **Overflow is failure.** If the canvas validator returns ‚öÝÔ∏è OVERFLOW, *cut content* ‚Äî do not shrink type below 24 px and do not push past 96 px margins. Prioritize: keep the TL;DR, keep the hero visual, drop the weakest finding card.
- **Audience-aware framing.** If the user's instructions or sources suggest a regulated / enterprise / finance context, lean into language like *control*, *exposure*, *threshold*, *owner*, *next action*, *committee*.

## Process

1. `source_list_files`, then read the heaviest signal sources. Pull real numbers, dates, and named entities.
2. **Choose body layout (a / b / c)** based on the content shape:
   - Findings + hero metric/chart ‚Üí **a**.
   - Long-form narrative with one strong image or hero stat ‚Üí **b**.
   - A framework, process, maturity model, or comparison ‚Üí **c**.
3. **Draft the four region contents** in a scratchpad before writing HTML ‚Äî title, TL;DR sentence, body items, recommendation, sources. Lock these *before* touching CSS.
4. **Write `styles/theme.css` first.** Declare `--bg`, `--ink`, `--ink-muted`, `--accent`, type tokens (`--text-tldr`, `--text-finding`, `--text-hero`, etc.), spacing tokens, the grid template for the body region, and reusable card styles. Make tokens generous and reuse them.
5. **Write `slides/slide1.html`** ‚Äî the single slide file. Root with `<section class="slide onepager">`; use semantic regions inside (`<header>`, `<main>`, `<footer>`). The `slide` class is mandatory (the canvas validator hooks into it); the `onepager` class is your own hook for layout-specific CSS.
6. **Validate.** If the canvas check passes cleanly, stop. If it flags overflow, cut and reshape; do not chase further "polish" once the page lands within bounds.

## One-pager layout ‚Äî the four-row grid

A one-pager has four regions competing for one 1080px canvas. To prevent the footer from ending up on top of the body, lay out `.onepager` as an explicit four-row CSS Grid:

```css
.onepager {
  display: grid;
  grid-template-rows: 160px 80px 1fr 130px;  /* header ¬∑ tldr ¬∑ body ¬∑ footer */
  grid-template-areas: "header" "tldr" "body" "footer";
  width: 1920px; height: 1080px;
  padding: 80px 96px;
  gap: 32px;
}
```

The body row is `1fr` so it absorbs leftover space; all others are pinned to px. The body is itself a sub-grid (e.g. `grid-template-columns: 1fr 1.1fr` for findings + visual). See the layout-discipline rules in the common-rules block above for the supporting techniques (`min-width: 0`, `overflow: hidden` per region, card `max-height` + `line-clamp`, no `position: absolute` in flow).

### When overflow recurs, cut in this order

The four-row grid is fixed; the body is the only flexible region. When the validator reports overflow on retry (especially the **same edge** as the previous attempt ‚Äî the framework will explicitly flag this), the total body weight is too heavy. **Cut, don't relocate.** Moving copy from `.op-actions` into `.op-footer` or from one finding card into another keeps the same vertical weight on the page ‚Äî the bottom edge will still overflow.

Cut in this priority:

1. **Right-edge overflow on `<header>`** ‚Äî the subtitle is too long for one line at the chosen font size. Shorten it (keep audience + date; drop scope qualifiers), or move the qualifiers into the optional `op-tag` chip.
2. **Bottom-edge overflow** ‚Äî work through this list until the body fits:
   1. Drop the optional `op-tag` chip if present.
   2. Trim the recommendation list (`.op-actions`) to 1‚Äì2 bullets (was 1‚Äì3).
   3. Apply `-webkit-line-clamp: 2` (with `display: -webkit-box; -webkit-box-orient: vertical; overflow: hidden`) to the so-what line on each finding card.
   4. Drop the weakest finding card ‚Äî 3 cards is fine; 5 was always borderline.
   5. Shrink the visual region from `1.1fr` to `0.9fr` in the body sub-grid so the findings column gets more room.
3. **Never** drop the TL;DR or the title ‚Äî they carry the page's argument.
4. **Never** shrink type below 24 px or push past the 96 px outer padding.

If you find yourself moving content between `.op-actions` and `.op-footer` across retries, you're not cutting ‚Äî you're reshuffling, and the total height won't change. Pick from the list above instead.
