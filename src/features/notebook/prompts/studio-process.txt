## Software / Process Lifecycle Frameworks

When the sources describe **how work flows through an organisation**, default to one of these established frameworks — pick the closest fit, do not invent a new one:

- **BPMN 2.0** — generic business processes. Use *start event* → *tasks* → *gateways (XOR / AND)* → *intermediate events* → *end event*. Use **pools** for organisations and **lanes** for roles within an organisation. Use **message flows** across pools, **sequence flows** within a pool.
- **Swimlane flowchart** — role-by-role view of a single process. One lane per role/team/system. Hand-offs become arrows crossing lane boundaries.
- **ITIL / ITSM** — change, incident, problem, request fulfilment. Lanes typically: Requester, Service Desk, Tech Team, CAB / Change Manager, Operations.
- **Software Development Lifecycle (SDLC)** — Requirements → Design → Build → Test → UAT → Release → Operate → Decommission. Lanes typically: Business, Product, Engineering, QA, Security/Compliance, Ops/SRE.
- **Three Lines of Defence** (banking risk) — lanes: 1st Line (Business), 2nd Line (Risk/Compliance), 3rd Line (Internal Audit). Use this when the source emphasises governance and oversight.
- **Porter Value Chain** — when the source is strategic / end-to-end and individual hand-offs aren't named. Lanes are *primary activities* (Inbound → Operations → Outbound → Marketing & Sales → Service) over *support activities*.
- **COBIT / control-flow** — when the source emphasises IT governance, segregation of duties, and control objectives.

If the sources do not fit any of the above, default to **BPMN-style swimlanes** — that is the most broadly understood notation in a finance-sector audit context.

## Modelling Rules — non-negotiable

1. **Exactly one start event.** A process begins with a single, named trigger ("Customer submits onboarding application", "Quarterly close kicked off"). No silent start.
2. **At least one explicit end event.** Multiple end events are fine and often necessary (happy path, reject path, exception path). Never leave a branch dangling.
3. **Every decision has labelled outgoing edges.** A decision node with two unlabelled arrows is a bug. Use the actual decision criterion as the label ("amount ≥ €100k", "KYC risk = high", "approved", "rejected").
4. **Decisions are framed as questions or conditions** ("Risk score ≥ threshold?", "Documentation complete?"). They must have **at least 2 outgoing edges**.
5. **Parallel split / join must be balanced.** Every `parallel` split has a matching `parallel` join further down the flow — do not leave parallel branches running off to different end events without joining.
6. **Tasks are verb-phrases naming an actor's action** ("Validate IBAN", "Approve credit limit", "Post journal entry"). Not "IBAN" — *verb the object*.
7. **One actor per task.** Place each task in the lane of the role/system that performs it. Hand-offs are edges crossing lane boundaries — they are the most valuable thing in the diagram, so make them explicit.
8. **Systems of record / data stores** ("Core Banking", "GL", "CRM") use the `data` kind, sit in their own *System* lane (or at the bottom), and are connected to the tasks that read/write them.
9. **Controls are first-class.** Where the source mentions a control (four-eye principle, segregation of duties, regulatory check, audit log), model it as its own `task` with `control` set to the framework reference (e.g. `"SOX 404"`, `"AML §11"`, `"BCBS 239 P3"`). Auditors look for these — do not bury them inside a generic "Review" task.
10. **Compactness over completeness.** Aim for **8–25 nodes total**. If a process is bigger, encapsulate detail using `subprocess` and note it in the description rather than expanding every step.
11. **Stable, descriptive ids.** Use short snake_case ids: `start`, `validate_iban`, `decision_kyc_risk`, `approve_credit`, `end_rejected`. Edges reference these. **Ids must be unique.**
12. **Design-first with traceability.** When the source is silent on a step, role, control, or exception branch, **fill it in** with what a senior BA would draw — don't leave the diagram incomplete. For each step you synthesised (not lifted from the source), **prefix its description with `(inferred)`** so the user can quickly tell what came from the documentation vs. your design proposal. This is the contract that keeps the output refinable and audit-honest.
